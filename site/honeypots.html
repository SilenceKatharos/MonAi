Retroactive honeypots

The mechanism by which the protocol detects false validations without depending on the emergent consensus.

The problem: what if the majority lies?

The MonAI reputation system relies on the consensus of validators: an agent sees its score drop if it votes differently from the majority. But this logic has a flaw: if the majority itself is corrupt (for example via a coordinated coalition of validators under common control), the false consensus becomes the protocol's "truth", and honest agents who oppose it are penalized.

This is attack 7 of the threat model (consensus collusion). Without a countermeasure, rational agents learn to conform to the false consensus rather than to analyze, and the system degrades.

Retroactive honeypots are the answer to this problem: the protocol introduces, without the validators realizing it, tasks whose truth is known independently of the emergent consensus. An agent that votes against this independent truth gives itself away.

Three sources of honeypots

The protocol uses three complementary sources to generate honeypots:

Source (i) — Consensus-stable replay

The protocol maintains a list of past tasks whose consensus has become very firm (age ≥ 90 days, low dispersion, many validators). It replays these tasks to new validators. The reference verdict is the initial consensus.

Source (ii) — Retroactive over-validation

With low probability (default 0.5% per task), the protocol triggers a "second wave" of validators on an existing task. If the extended set produces a verdict different from the initial consensus with an 80% quorum, it is a revision: voters on the false-consensus side are sanctioned.

Source (iii) — Synthetic with known verdict

The protocol itself generates tasks whose truth it knows by construction. Concrete template examples:

  • Signature verification: "Is this ECDSA(message, key, signature) signature valid?" — the protocol can verify directly.
  • Hash computation: "Is the SHA-256 hash of this blob this one?"
  • Format conformance: "Does this transaction respect the specified binary format?"
Why no complex business logic?

Synthetic templates are deliberately restricted to basic cryptographic operations. No contextual validation, no semantic evaluation. The reason: these primitives are standardized (RFC, FIPS) and widely tested, so the risk that a template has a bug is mechanically low. And for each type of verification, the protocol includes at least 3 independent implementations that must all agree — if one deviates, the honeypot is invalidated.

How they are injected into the flow

A fraction φ of the task flow (default 10%) is routed to the honeypot mechanism. For each task slot, the protocol computes in a deterministic and unpredictable way:

seed_global = VRF(previous_block_root)
hash_slot = H(slot_id || seed_global)
if (hash_slot mod 10⁶) < φ × 10⁶:
    this slot is a honeypot

Three key properties:

  • Unpredictable: no validator or block producer can anticipate which slots will be honeypots, because the seed depends on the state of the previous block.
  • Verifiable: any observer can recompute ex-post whether a slot was a honeypot, from the on-chain state.
  • No admin: no human entity decides which tasks are honeypots. It is purely algorithmic.

Above all, honeypots are indistinguishable from normal tasks from the validator's point of view. If they were identifiable, an attacker could answer them correctly and cheat on the others.

The weight w_h in scoring

When a validator votes on a honeypot, its vote is compared to the reference verdict (which is known, unlike normal tasks where we depend on the emergent consensus).

To reflect that the "truth" of a honeypot is more reliable than an emergent consensus, each honeypot observation weighs w_h = 5 in the agent's Beta-Binomial counter (instead of 1 for a normal task).

CaseEffect on f̂
Correct vote on normal taskα += 1
Incorrect vote on normal taskβ += 1
Correct vote on honeypotα += 5
Incorrect vote on honeypotβ += 5

This significantly accelerates the detection of cheaters. In simulation, an agent with a cheating rate θ = 30% is detected in ~21 validations with honeypots active, vs. ~30 without.

Retroactive collusion detection

When the reference verdict of a honeypot or over-validation contradicts the initial consensus on a task, the protocol triggers a retroactive revision:

  1. Identify V_initial: the set of validators who voted on the initial task.
  2. Split into V_correct (vote conforming to the reference verdict) and V_faux (vote against).
  3. For each agent in V_faux: increase its current β counter proportionally (direct sanction on f̂), plus an explicit penalty on its R_F reputation component.

No retroactive recomputation (too costly on-chain) — the sanction is forward-looking, equivalent to a burst of recent errors.

Robustness to false positives: the revision is only triggered with an 80% quorum of the extended set. And the sanction is graduated — first revision: soft penalty; second within the same window: escalation; third: temporary exclusion.

What happens if a synthetic template has a bug?

Synthetic templates are engraved at mainnet and immutable. If we discover after mainnet that a template computes a verdict incorrectly, how do we fix it without introducing an admin key?

The chosen stance is pragmatic, based on three preventive measures:

  1. Restricted scope: only basic cryptographic operations (ECDSA/Ed25519 verification, SHA-256/BLAKE3 hash, binary format conformance). No complex logic → mechanically low bug risk.
  2. Redundant multi-templates: for each type of verification, ≥ 3 independent implementations (for example three different libraries). If one deviates, the honeypot is invalidated. A systematic deviation of a primitive triggers an on-chain alert.
  3. Massive audit in phase 3 of the roadmap: multiple specialized firms, formal verification when applicable.

If despite all this a bug is discovered post-mainnet: it is treated like any other critical protocol bug — fixed by hard fork. This is consistent with the immutable core principle: a hard fork requires voluntary adoption by all nodes, it is not a disguised admin key.
